# Architecture Overview

## Scope of the Current Implementation

The current repository snapshot implements the service and shared-model layers of AccessMesh-AI. It does not yet include an active API runtime, frontend application, or infrastructure provisioning files.

This means the architecture already has the building blocks for an accessibility pipeline, but not yet the orchestration entrypoints that would expose those blocks as a running product.

## Architectural Layers

### 1. Shared Contracts

The shared layer defines the common data model used by the rest of the system.

- `shared/config.py`
  - Central settings registry.
  - Supports environment variables, `.env`, and Azure Key Vault.

- `shared/message_schema.py`
  - Defines the message envelope used across the intended pipeline.
  - Standardizes message shape, metadata, timestamps, sender, session, and message type.

### 2. Integration Services

The `services/` folder contains focused wrappers around Azure capabilities and supporting platform concerns.

- Real-time messaging: Web PubSub.
- Async transport: Service Bus.
- Persistence: Cosmos DB.
- Speech: token issuance and transcription.
- Translation: Azure Translator.
- Summarization: Azure OpenAI.
- Gesture recognition: rule-based + Azure OpenAI.
- Safety: Azure Content Safety.
- Voice output: Azure Speech Neural TTS.
- Secrets: Azure Key Vault.
- Observability: Application Insights / OpenTelemetry.

### 3. Orchestration Layer

This layer is implied by the repository structure and message design, but is not yet implemented in the current workspace.

Expected responsibilities:

- Receive input from clients.
- Convert input into shared message models.
- Dispatch processing through services.
- Persist events and state.
- Broadcast outputs to connected clients.
- Route async work through Service Bus when needed.

## Intended End-to-End Flow

### Real-time communication flow

1. Client joins a session.
2. Client requests a Web PubSub token.
3. Client sends audio, text, or gesture data.
4. Backend transforms incoming payloads into shared message objects.
5. Processing services enrich or transform the content.
6. Outputs are stored and then delivered in real time.

### Accessibility flow

1. Speech input is transcribed by `SpeechService`.
2. Gesture input is interpreted by `GestureService`.
3. Text may be checked by `ContentSafetyService`.
4. Text may be translated by `TranslatorService`.
5. Voice output with visemes may be produced by `AvatarService`.
6. A meeting summary may be generated by `SummarizationService`.

### Persistence flow

`CosmosService` persists:

- Sessions.
- Messages.
- Users.

This supports replay, history, summaries, participant context, and future analytics.

### Messaging flow

The repository indicates two complementary messaging patterns:

- `WebPubSubService` for low-latency client fan-out.
- `ServiceBusService` for asynchronous and decoupled backend processing.

This split is appropriate because real-time client delivery and reliable backend transport solve different problems.

## Message Model Design

The message schema already reflects a multi-stage processing pipeline.

Core message types include:

- `audio_chunk`
- `transcription`
- `gesture`
- `routed`
- `accessible`
- `translated`
- `avatar_ready`
- `summary`
- `system`
- `error`

This design suggests a pipeline where each stage receives a typed message and emits a new typed message after transformation.

## Service Responsibilities

### WebPubSubService

- Real-time hub integration.
- Token issuance.
- Broadcast to all, groups, or users.
- Group membership management.

### SpeechService

- Client speech token generation.
- Server-side transcription from raw audio bytes.

### GestureService

- Label-to-text normalization.
- Rule-based hand landmark classification.
- Vision-based recognition with Azure OpenAI.

### TranslatorService

- Text translation.
- Optional source-language handling.

### SummarizationService

- Transcript summarization.
- Extraction of key points.

### AvatarService

- TTS synthesis.
- Viseme generation for avatar animation.
- Placeholder for future sign-language synthesis.

### CosmosService

- Database and container bootstrap.
- CRUD-style async operations for core entities.

### ServiceBusService

- Topic publishing.
- Subscription receiver creation.
- Transport abstraction for decoupled workers.

### ContentSafetyService

- Moderation gate for text.
- Threshold-based decision making.

### KeyVaultService

- Secret retrieval and writing.
- Managed authentication through Azure Identity.

### TelemetryService

- Span-based tracing.
- Custom event tracking.
- Application Insights integration.

## Design Characteristics Already Present

The current implementation already shows several sound design choices:

- Clear separation of concerns.
- Configuration-driven service bootstrapping.
- Support for disabled or stub modes when credentials are absent.
- Consistent logging.
- Strongly typed message contracts.
- Async support where external I/O is expected.
- Azure-first integration model.

## What Is Missing to Make It Runnable End-to-End

To turn the current code into a complete running application, the project still needs:

1. An API layer, most likely in `backend/`.
2. Request/response endpoints and WebSocket or PubSub negotiation endpoints.
3. Orchestration code connecting all services together.
4. A UI in `frontend/`.
5. Deployment assets in `infrastructure/`.
6. Integration tests and operational documentation.

## Practical Interpretation

This repository is best understood as a platform foundation. The core integrations and domain contracts are already implemented, and they strongly indicate the product direction. What is still pending is the application shell that ties them together into a deployable user-facing system.
